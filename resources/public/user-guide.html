Bitool User Guide

From business intent to verified data pipeline in minutes, not months. This guide covers every feature of the Bitool BI platform end-to-end.

What is Bitool?

Bitool is a full-stack business intelligence platform that automates the entire data pipeline lifecycle: connecting to data sources, ingesting raw data (Bronze), cleaning and standardizing (Silver), building analytics models (Gold), and querying via AI-powered natural language reporting — all governed by a production-grade semantic layer.

Core Philosophy The graph is the source of truth. SQL is a compiled artifact. AI proposes — deterministic compilers validate — humans verify visually. No hallucinated SQL ever reaches your database.
Business Intent (plain English) | v Visual Graph (editable, inspectable) | v Compiled SQL (deterministic, auditable) | v Warehouse Execution (PostgreSQL, Snowflake, Databricks, BigQuery)

Bronze Layer (Raw Ingestion)

The Bronze layer captures raw data from any source into your warehouse with full fidelity. Every record is wrapped in a standard envelope with 16 system columns for lineage, replay, and auditing.

Bronze Envelope Columns

ColumnPurpose

Running an Ingestion

  1. Select a configured source (API endpoint, file, Kafka topic)
  2. Review the auto-inferred schema — approve or adjust column mappings
  3. Choose ingestion mode: Full (complete re-extract) or Incremental (watermark-based)
  4. Click Run — Bitool streams records in bounded batches with adaptive backpressure

Schema Approval

When Bitool first encounters a new API or file source, it infers the schema automatically. Before data reaches Bronze, the schema goes through an approval flow:

  1. Preview: View the inferred schema with data types and sample values
  2. Review: Approve, reject, or modify column names and types
  3. Promote: Approved schemas are promoted to the Bronze layer definition

Bad Record Handling

Records that fail schema validation or parsing are quarantined in a dead letter queue (DLQ), not dropped silently. You can:

  • View bad records in Operations > Bad Records
  • Inspect the failure reason for each record
  • Fix the schema or source, then Replay the bad records

Silver Layer (Cleaning & Standardization)

Silver transforms raw Bronze data into typed, deduplicated, business-ready entities. Bitool auto-proposes Silver schemas using AI — you review and approve.

Auto-Proposal

Click Propose Silver Schema on any Bronze table. Bitool analyzes the raw data and proposes:

  • Entity classification: fact, dimension, reference, or snapshot
  • Column roles: business_key, timestamp, measure, attribute, dimension
  • Grain: The natural key that defines one row (e.g., customer_id + order_date)
  • Primary keys: Inferred from uniqueness analysis
  • Deduplication rules: Merge logic for handling duplicates
  • Type casting: Raw strings converted to proper types (dates, numbers, booleans)
Confidence Scores Every proposal carries a confidence score (0–1). Business key matches score 0.95, schema passthrough 0.85, count fallback 0.82. Low confidence = human review needed.

Review & Execute

  1. Review the proposal — click any field to override the AI suggestion
  2. Click Validate to check the proposal against your warehouse (DDL compatibility)
  3. Click Publish then Execute to materialize the Silver table

Gold Layer (Analytics Models)

Gold builds business-ready analytical models from Silver entities. These are the tables that power reports, dashboards, and the semantic layer.

Auto-Proposal

Click Propose Gold Schema to auto-generate:

  • Aggregated measures: SUM(revenue), COUNT(*), AVG(trip_miles)
  • Time dimensions: Event dates bucketed by day, week, month, quarter, year
  • Dimension keys: Group-by columns for slicing and dicing
  • Mart tables: Pre-joined, denormalized views optimized for BI consumption

BRD-Driven Gold

For more complex analytics, upload a Business Requirements Document (BRD) and click Generate Gold from BRD. Bitool reads the requirements and proposes a complete Gold layer with measures, dimensions, and grain matching the stated business need.

Example Prompt "Connect to Samsara's fleet API. Pull vehicle locations, driver activity, and fuel transactions. Build a daily fleet utilization model and a driver safety scorecard."

Bitool generates:

You can then edit any part of the generated pipeline visually or via text, and the system recompiles deterministically.

Text-to-Graph (GIL)

The Graph Intent Language (GIL) is the intermediate representation for all graph mutations. It lets you describe graph changes in English and have them compiled into visual graph nodes.

Workflow

  1. Click GIL > From Natural Language in the toolbar
  2. Type your intent: "Join trips with drivers on driver_id. Filter to this month. Group by region and sum miles."
  3. Bitool generates a GIL document (structured intent), validates it, and compiles it into graph nodes
  4. Review the generated graph visually — edit any node if needed
  5. Click Apply to commit the changes to your graph
LLMs propose, compilers validate GIL uses the same hallucination-prevention pattern as ISL: the AI generates a constrained intent document, a deterministic compiler validates every reference against the actual schema, and only valid mutations reach the graph. Invalid proposals are rejected with a clear error message.

GIL Pipeline

Natural Language → GIL Document → Normalize → Validate → Compile → Apply to Graph

GIL handles node typing, graph structure, planning, persistence, and graph application. The canonical-type function maps semantic node names to internal btype codes (e.g., "function" → "Fu"), and valid-edge? enforces connectivity rules. Layout is computed automatically.

NL → Transform Graphs

Beyond simple query-oriented graph mutations, Bitool supports generating complete transform graphs from natural language. This is the transform-side equivalent of the AI Reporting pipeline: instead of NL → ISL → SQL, it's NL → constrained transform GIL → graph → runtime execution.

What you can say "Create an API flow that normalizes customer input, assigns a risk score, branches on risk level, and returns a decision payload."
"Build an order transform that cleans fields, computes discounts, routes approvals, and shapes a final response."
"Generate a transform graph that validates input, routes priority orders, and returns JSON."

How It Works

User NL → parse-text (LLM or deterministic fallback) → Semantic Plan → semantic-plan→GIL (constrained transform GIL) → validate-transform-gil (topology + data flow) → Preview (data/text) → Apply → Graph on Canvas
  1. Open the Transform Chat panel
  2. Describe the transform you want in plain English
  3. The system calls the configured LLM provider (Anthropic first, OpenAI fallback), validates the generated GIL, and attempts a structured repair pass if the output is invalid
  4. If the LLM output is unusable, a deterministic fallback library of 5 domain templates is used (order/customer, invoice, lead, customer mastering, eligibility)
  5. Preview the generated graph with validation status
  6. Click Apply Graph to render nodes on the canvas

Supported Transform Patterns

V1 supports linear transform chains using Logic and Conditional nodes:

PatternUse Case
Endpoint → Logic → Conditional → Logic → ResponseBuilder → OutputAPI flow: normalize, branch, compute, respond
Endpoint → Conditional → Logic → ResponseBuilder → OutputRoute input to different processing paths
Endpoint → Logic → Logic → ResponseBuilder → OutputMulti-stage data transformation
Table → Logic → Conditional → Logic → OutputBatch transform with conditional routing

Logic Nodes (Fu)

Logic nodes support ordered assignments, expression evaluation, and multiple outputs. Each node has three phases:

  • fn_params — binds input variables from upstream data
  • fn_lets — ordered assignment rows (each can reference params + earlier assignments)
  • fn_outputs — the values passed downstream

Conditional Nodes (C)

Conditional nodes support multiple evaluation styles:

  • if-else — simple binary branching
  • if-elif-else — ordered guard clauses
  • case — exact-match switching
  • cond — general condition list
  • pattern-match — ordered pattern guards

Expression Language

Transform expressions support a rich set of built-in functions. The recommended v1 subset includes:

trim, upper, lower, concat, tonumber, toboolean, tostring, coalesce,
contains, startswith, endswith, replace, round, length, if, isnull,
isempty, min, max, sum

Operators: arithmetic (+ - * / %), comparison (= != < > <= >=), boolean (&& || ! and or not).

Validation & Safety

  • Topology validation: Exactly 1 source, 1 output, linear chain only (no branching topologies in v1)
  • Data flow validation: Every source_column reference must point to a known upstream stage output
  • Repair loop: If the LLM generates invalid GIL, the system attempts one structured repair pass before falling back to deterministic templates
  • No arbitrary code: Execution runs through the existing Clojure runtime, never through LLM-generated source files

POST/transform/applyApply generated transform to canvas

Before AI Reporting can answer questions about your data, it needs business context — what each table and column means. Bitool offers two complementary training paths.

1. Schema Fingerprinting (Zero-LLM)

Bitool auto-recognizes 8 common SaaS schemas without any LLM calls:

SourceDetection Method

HubSpotMarker tables: contact, deal, company, etc.

Detection requires 2+ marker table matches. Handles Fivetran naming (charge), prefixed naming (stripe_charges), and double-underscore (stripe__charges) patterns. Detected sources appear as badges in the reporting sidebar.

2. Auto-Train (LLM-Assisted)

For tables not recognized by fingerprinting:

  1. Click Auto-Train in the reporting panel (or call POST /api/reporting/auto-train)
  2. Bitool samples 3 rows from each table
  3. The LLM generates business descriptions for each table and column
  4. Descriptions are stored in the schema_context table and used to enrich future AI prompts
Curated descriptions take priority Fingerprinted descriptions (source = "fingerprint") are checked before LLM generation. This means recognized SaaS schemas get instant, accurate context with zero LLM cost.

Manual Training

You can also manually add or edit business descriptions:

  • Table-level: "This table contains all Stripe payment charges"
  • Column-level: "amount — charge amount in cents (USD)"

API: POST /api/reporting/context/saveEntry to create/update, DELETE /api/reporting/context/id/:context_id to remove.

Zero-Config AI Reporting

Connect your warehouse — PostgreSQL, MySQL, Oracle, SQL Server, DB2, Snowflake, or Databricks — ask any question, and get a validated answer in seconds. No setup, no analyst, no dashboard to build.

How It Works

User Question (English) | v Dynamic Tool Schema (from registry/semantic model) | v LLM generates ISL (constrained intent, not raw SQL) | v validate-isl (deterministic check against schema) | v compile-isl (ISL to SQL, with measure expansion) | v Execute on warehouse → Return results

Key Guarantees

  • No hallucinated SQL: The LLM generates a constrained ISL document, not free-form SQL. Every table, column, and join is validated before compilation.
  • Enum-locked tables: The LLM can only reference tables that actually exist in your schema.
  • Semantic measures: When a semantic model is published, calculated measures (e.g., cost_per_mile) become enum-locked options the LLM can pick from.
  • Follow-up context: Ask follow-up questions — Bitool maintains conversation context.

Scoping Queries

You can scope the AI to specific tables by setting a table filter. This limits what the LLM sees and speeds up responses:

POST /api/reporting/ask
{
  "conn_id": 42,
  "schema": "analytics",
  "question": "Top 10 drivers by fuel cost this month",
  "scope_tables": ["gold_trips", "silver_drivers"],
  "perspective": "finance"
}

ISL Pipeline Details

ISL (Intent-Specified Language) is the intermediate representation between natural language and SQL. An ISL document looks like:

{
  "table": "gold_trips",
  "columns": ["driver_id", "drivers.region"],
  "aggregates": [
    {"column": "fuel_cost", "function": "SUM"},
    {"column": "cost_per_mile", "function": "AVG"}
  ],
  "filters": [
    {"column": "event_date", "op": ">=", "value": "2026-01-01"}
  ],
  "group_by": ["driver_id", "drivers.region"],
  "order_by": [{"column": "SUM(fuel_cost)", "direction": "DESC"}],
  "limit": 10
}

When a semantic model is active, the ISL pipeline automatically:

  • Injects lazy joins: Cross-entity refs like drivers.region trigger automatic JOIN resolution via BFS shortest-path
  • Applies perspectives: Filters the model to the audience's allowed entities/columns/measures
  • Enforces RLS: Injects row-level security filters based on the user's session
  • Expands measures: Semantic measures like cost_per_mile are expanded to their SQL expressions at compile time

Saved Reports

Save any query result as a named report for re-execution later:

  1. After a successful query, click Save Report
  2. Enter a name and optional description
  3. The ISL, semantic model ID, and perspective are persisted together
  4. Re-run the report at any time — it restores the full semantic context (model, perspective, RLS)
Semantic Binding Saved reports carry semantic_model_id and perspective_name. When re-executed, the original semantic model and perspective are restored, ensuring consistent results even as the underlying data changes.

Semantic Layer Overview

The Semantic Layer is a named, versioned, governed business layer between raw tables and reports. It bundles everything the ISL pipeline needs into a single object: entities, relationships, measures, hierarchies, and business descriptions.

Why It Matters

Without a semantic layer, every report rebuilds context from scratch: which tables to use, how they join, what measures mean. The semantic layer captures this once and enforces it everywhere:

  • AI reporting uses entity names and measure definitions instead of raw table/column names
  • Joins are resolved automatically from declared relationships (lazy associations)
  • Calculated measures are expanded at compile time
  • Perspectives control what each audience can see
  • Row-level security filters are enforced transparently

Auto-Generate a Semantic Model

Bitool infers 80% of the semantic layer from existing Silver/Gold proposals. No manual YAML or LookML required.

1

Generate

POST /api/semantic/generate with conn_id and graph_ids. Bitool assembles entities, columns, relationships, measures, and descriptions from existing proposals.

2

Review

The model starts as draft. Browse entities, check column roles, verify relationships. Edit anything that needs adjustment.

3

Validate

POST /api/semantic/models/:id/validate checks: expression syntax, column dependencies, measure cycles, relationship uniqueness.

4

Publish

POST /api/semantic/models/:id/publish transitions to published. AI reporting now uses this model automatically.

Model Structure

{
  "entities": {
    "trips": {
      "kind": "fact",
      "table": "gold_trips",
      "columns": [
        {"name": "trip_id",    "role": "dimension", "type": "BIGINT"},
        {"name": "miles",      "role": "measure",   "type": "DOUBLE"},
        {"name": "fuel_cost",  "role": "measure",   "type": "DOUBLE"},
        {"name": "event_date", "role": "time_dimension", "type": "DATE"}
      ]
    },
    "drivers": {
      "kind": "dimension",
      "table": "silver_drivers",
      "columns": [
        {"name": "driver_id",   "role": "business_key"},
        {"name": "driver_name", "role": "attribute"},
        {"name": "region",      "role": "attribute"}
      ]
    }
  },
  "relationships": [
    {"from": "trips", "from_column": "driver_id",
     "to": "drivers", "to_column": "driver_id",
     "type": "many_to_one", "join": "LEFT"}
  ],
  "calculated_measures": [
    {"name": "cost_per_mile",
     "expression": "fuel_cost / NULLIF(miles, 0)",
     "entity": "trips"}
  ],
  "restricted_measures": [
    {"name": "emea_fuel_cost",
     "base_measure": "SUM(fuel_cost)",
     "filter_column": "region",
     "filter_values": ["EMEA"],
     "entity": "trips"}
  ],
  "hierarchies": [
    {"name": "geography", "entity": "drivers",
     "levels": [{"column": "region"}, {"column": "driver_name"}]}
  ]
}

Measures & Hierarchies

Calculated Measures

Define business metrics as expressions over entity columns:

POST /api/semantic/models/:id/measures/calculated
{
  "name": "cost_per_mile",
  "entity": "trips",
  "expression": "fuel_cost / NULLIF(miles, 0)",
  "description": "Fuel efficiency metric"
}

Supported expression language:

  • Arithmetic: +, -, *, /
  • Functions: NULLIF, COALESCE, ABS, ROUND, CAST
  • Aggregates: SUM, COUNT, AVG, MIN, MAX (post-aggregate measures)
  • References: Bare column names resolve to the entity's columns or other calculated measures
Dependency Ordering Calculated measures may reference other calculated measures. Bitool topologically sorts them and rejects cycles. At compile time, nested references are expanded recursively.

Restricted Measures

Filtered aggregates that apply a WHERE condition automatically:

POST /api/semantic/models/:id/measures/restricted
{
  "name": "emea_fuel_cost",
  "entity": "trips",
  "base_measure": "SUM(fuel_cost)",
  "filter_column": "region",
  "filter_values": ["EMEA"]
}

Hierarchies

Define drill paths for dimensional analysis:

POST /api/semantic/models/:id/hierarchies
{
  "name": "geography",
  "entity": "drivers",
  "levels": [
    {"column": "region"},
    {"column": "driver_name"}
  ]
}

Perspectives (Audience-Scoped Views)

Perspectives are named subsets of a semantic model that control what a particular audience sees. The finance team sees revenue metrics; ops sees fleet utilization; executives see both but with fewer detail columns.

Creating a Perspective

POST /api/semantic/models/:id/perspectives
{
  "name": "finance",
  "audience": "finance_team",
  "description": "Revenue and cost metrics for finance",
  "spec": {
    "entities": ["trips", "drivers"],
    "columns": {
      "trips": ["miles", "fuel_cost", "region", "event_date"]
    },
    "measures": ["cost_per_mile", "emea_fuel_cost"],
    "hierarchies": ["geography"]
  }
}

Any key omitted or null means "no restriction" (allow all). When a perspective is active:

  • Only the listed entities, columns, measures, and hierarchies are visible
  • Relationships between excluded entities are removed
  • The AI reporting tool schema is scoped accordingly
Consistency Validation Bitool validates that retained measures and hierarchies still have their column dependencies after perspective filtering. A perspective that hides miles but keeps cost_per_mile (which divides by miles) is rejected with a clear error.

Using a Perspective in Queries

POST /api/reporting/ask
{
  "conn_id": 42,
  "question": "What is our cost per mile by region?",
  "perspective": "finance"
}

Preview Without Saving

POST /api/semantic/models/:id/perspectives/preview
{
  "spec": {
    "entities": ["trips"],
    "measures": ["cost_per_mile"]
  }
}

Lazy Associations (Automatic Joins)

Relationships in the semantic model are lazy — they're defined once and only resolved when a query references a column from the associated entity. This eliminates unnecessary joins and matches SAP Datasphere's association pattern.

How It Works

  1. You define relationships: trips.driver_id → drivers.driver_id (many_to_one LEFT)
  2. A user asks: "Total miles by driver region"
  3. The ISL references drivers.region — a column on the drivers entity
  4. The lazy join resolver detects this cross-entity reference
  5. BFS shortest-path finds the join route: trips → drivers
  6. A LEFT JOIN is injected into the ISL automatically

Ambiguity Resolution

When multiple join paths of equal length exist (e.g., A → B → D and A → C → D), Bitool:

  • Detects all shortest paths via layer-by-layer BFS
  • If one path has all edges marked :preferred true, uses that path
  • Otherwise, rejects with an ambiguity error listing the candidates

Row-Level Security (RLS)

RLS policies restrict query results based on the user's session attributes. They are enforced transparently at ISL query time — no code changes needed in reports.

Creating an RLS Policy

POST /api/semantic/models/:id/rls
{
  "entity": "drivers",
  "column_name": "region",
  "user_field": "region",
  "allowed_values": ["EMEA", "NA", "APAC"]
}

This policy means: when a user whose session has region = "EMEA" runs a query, a WHERE silver_drivers.region = 'EMEA' filter is injected automatically. If their session value is NOT in allowed_values, the query is blocked with a 403.

How RLS is Enforced

  1. User submits a question via /api/reporting/ask
  2. After lazy join injection, the pipeline checks RLS policies for the active model
  3. For each policy, it checks the user's session for the user_field
  4. If the user's value is in allowed_values, a WHERE filter is injected into the ISL
  5. If not in allowed_values or the session field is missing, the request is blocked
  6. The ISL is then validated and compiled with the RLS filters included
Transparent Enforcement RLS filters are injected server-side before SQL compilation. The user never sees the filter in their question, and cannot bypass it. Even saved reports re-apply RLS on execution.

Alert Lifecycle

Alerts follow a lifecycle: fired → acknowledged → silenced or resolved. The Ops Console shows all active alerts with severity levels and remediation suggestions.

Schema Drift Detection

When upstream sources change their schema (new columns, removed columns, type changes), Bitool detects the drift automatically and:

  1. Raises a drift alert with severity (added columns = low, removed = high, type change = medium)
  2. Shows a diff between the expected and actual schema
  3. Suggests remediation actions using AI (/aiSuggestDriftRemediation)
  4. Allows you to approve the drift (update the schema) or reject it (keep the old schema and quarantine new records)

Checkpoints & Replay

Every ingestion maintains a watermark checkpoint so it can resume exactly where it left off:

  • Verify: Check that all checkpoints are consistent and no gaps exist
  • Reset: Roll back a checkpoint to re-ingest from a specific point
  • Replay: Re-process a specific batch or set of bad records
  • Rollback: Remove a batch's data from the warehouse and restore the previous checkpoint

Endpoint Authoring

Create REST APIs directly from your graph transformations:

  1. Add an Endpoint node to your graph with URL path and parameters
  2. Connect it to transform nodes (Filter, Join, Aggregation, etc.)
  3. Add a Response Builder to shape the output (JSON structure)
  4. Optionally add a Validator for input parameter validation
  5. Click Deploy Endpoints to register the API routes

Deployed endpoints are live REST APIs that execute the graph on each request, with parameters flowing from the URL/body into the Endpoint node's columns.

Multi-Tenancy & Workspaces

Bitool supports multi-tenant operation through workspaces:

  • Tenants: Top-level isolation (separate organizations)
  • Workspaces: Sub-divisions within a tenant (dev, staging, prod)
  • Members: Users assigned to workspaces with roles (viewer, editor, admin)
  • Graph assignment: Graphs and semantic models can be scoped to specific workspaces
  • Promotion: Move published models between workspaces (dev → prod)

## 1. Product Vision

Bitool is the complete data platform. Everything else is a component looking for a product.

Users connect a warehouse, the system auto-recognizes what's there, and they ask questions in plain English. No semantic models, no YAML configs, no DE team. The conversation is the product.

**Core thesis**: The graph is the source of truth — not SQL, not drag-drop UI, not LLM-generated code. LLMs propose, the deterministic compiler decides, humans verify visually.

### 1.1 Strategic Position: Be the Replacement, Not Additive

Bitool is not another tool in the stack. It is the stack.

| Instead of... | Bitool answer |
|---|---|
| "We integrate with dbt" | "Why do you need dbt?" |
| "We export to Cube" | "Cube is what you build when you don't have Bitool" |
| "We're compatible with Informatica" | "We're what replaces your Informatica contract" |
| "We work alongside Fivetran + Tableau" | "We replace both — and the DE team connecting them" |

dbt is a component. Cube is a component. Informatica is legacy. Fivetran is a truck. Power BI is a Microsoft accessory. Looker/Tableau bolt LLM onto dashboards and still require DE teams for the semantic layer.

### 1.2 The Full-Stack Moat

The iPhone's deepest moat wasn't the touchscreen — it was that every layer reinforced every other layer. Bitool's equivalent:

```
Pipeline built in Bitool
    -> transforms defined in Bitool
        -> semantic layer owned by Bitool
            -> reports generated by Bitool
                -> ops monitored by Bitool
                    -> lineage, governance, alerts — all Bitool
```

Every BI cycle stage (Connect → Ingest → Schedule → Transform → Model → Train → Report → Ops → Self-heal → Governance → Testing) is owned by the platform. Every layer a customer uses makes leaving harder. That's the moat — not exports, not integrations.

### 1.3 Data Portable, Workflow Locked

> "Your data stays in your Snowflake/Databricks/GCP. We don't lock your data. We lock your workflow — because our workflow is better."

Bitool never stores customer data — it stays in the customer's warehouse. What we own is the pipeline, the transform logic, the semantic layer, the reports, and the operational intelligence. Data portability: yes. Workflow portability: nobody asks for an "export my muscle memory" feature.

## 4. Connect — Source Discovery & Fingerprinting

### 4.1 Connector System

**Files**: `connector/api.clj`, `connector/auth.clj`, `connector/paginate.clj`, `connector/pagination.clj`

- Connection-pooled HTTP client (64 threads, 16 per-host)
- Auth: Bearer, API-key, Basic, OAuth2
- Pagination strategies: offset, page, cursor, token, time-based, Link header (RFC 5988)
- Async fetching via core.async with backpressure and rate limiting
- GraphQL support: introspection, connection-page pagination

### 4.2 OpenAPI Integration

**Files**: `connector/schema.clj`, `connector/apid.clj`, `api/openapi.clj`

- Parse OpenAPI v3 and Swagger v2 specs
- Auto-detect pagination strategy from spec
- Convert spec → graph nodes (table source nodes with typed columns)
- `$ref` dereferencing, auth scheme extraction

### 4.3 File Sources

**File**: `connector/file.clj`

- CSV, JSON, Parquet, COBOL Copybook parsing
- Async file reader with checksum verification

### 4.4 Kafka Sources

**File**: `connector/kafka.clj`

- Native consumer with async fetching
- Schema registry integration

### 4.5 Dynamic Endpoints (Ep/Wh)

**File**: `endpoint.clj`

- Runtime endpoint registry: `{graph-id {node-id {:method :path :config}}}`
- Ep nodes registered at startup via `restore-endpoints!`
- Webhook signature verification (HMAC-SHA256)
- Ep params become query columns via `ep-params->tcols`

### 4.6 Connection Management

**File**: `db.clj`

- `save-conn`, `test-connection`, `list-all-connections`, `get-connection-detail`
- Schema introspection: `introspect-schema`, `get-schema-registry`, `discover-joins`
- Multi-database support: `get-databases`, `get-schemas`, `get-tables`, `get-columns`

### 4.7 Schema Fingerprinting

**File**: `schema_fingerprints.clj`

Auto-recognizes known SaaS schemas (Fivetran/Airbyte output) without LLM calls.

**Catalog**: 8 sources — Stripe, HubSpot, Salesforce, Shopify, QuickBooks, GA4, Jira, Zendesk

**Detection**: `detect-sources` matches table names against marker-table sets (need 2+ matches). Handles Fivetran naming (`charge`), prefixed naming (`stripe_charges`), and double-underscore (`stripe__charges`) via `normalize-table-name`.

**Integration**:
- `reporting-auto-train` checks curated descriptions before LLM (source="fingerprint")
- `reporting-schema-response` returns `detected_sources` array for UI badges
- `curated-descriptions-for-registry` provides bulk lookup for all tables in scope

## 5. Ingest — Bronze Layer (Raw Ingestion)

**Purpose**: Ingest raw data from APIs, files, Kafka, webhooks into warehouse tables with full lineage.

**Key files**: `ingest/runtime.clj`, `ingest/execution.clj`, `ingest/bronze.clj`, `ingest/checkpoint.clj`, `ingest/schema_infer.clj`

### 5.1 Execution Model

```
enqueue-*-request! -> execution_request (queued)
       |
process-next-request! -> lease -> running
       |
  +----+----+
  | success | -> completed, checkpoint advanced
  | failure | -> retry (transient) or DLQ (permanent)
  +---------+
```

- **Request types**: API, Scheduler, Kafka, File, Silver release, Gold release
- **Deduplication**: Hash-based on active statuses prevents duplicate requests
- **Retry classification**: `transient_network`, `rate_limited`, `target_conflict` → retryable; others → DLQ
- **Orphan recovery**: Detects leaked leases, re-queues

### 5.2 Schema Inference

**File**: `ingest/schema_infer.clj`

- `detect-dominant-records-path` — finds array containing records (e.g., `$.data`, `$.items`)
- `infer-fields-from-records` — produces field descriptors
- `effective-field-descriptors` — merges inferred + manual + OpenAPI schema

**Enforcement modes**: strict (reject unknown), additive (accept + track), permissive (accept all)

### 5.3 Bronze Row Structure

`promoted-columns` added by Bitool: `bitool_sysid`, `bitool_watermark`, `bitool_sequence`, `bitool_source`, `bitool_batch_id`, etc.

### 5.4 Checkpoint and Watermark

**File**: `ingest/checkpoint.clj`

- Global checkpoint = only committed truth (monotonic)
- Window-based (hourly/daily) watermark progression

### 5.5 Bad Records

- Quarantined with metadata (reason, source batch, original payload)
- Replay API supports best-effort recovery

## 6. Schedule — Cron for Recurring Runs

**Key file**: `ingest/scheduler.clj`

### 6.1 Scheduler Node (Sc btype)

The Scheduler node type (`Sc`) allows users to attach cron schedules to any graph. Defined in the graph editor via `schedulerComponent.js`.

### 6.2 Cron Engine

**File**: `ingest/scheduler.clj`

- Full cron expression parser: minute, hour, day-of-month, month, day-of-week
- Supports ranges (`1-5`), steps (`*/15`), lists (`MON,WED,FRI`), named tokens (`JAN`, `SUN`)
- `scheduler_run_state` table tracks: `graph_id`, `scheduler_node_id`, `scheduled_for_utc`, `triggered_at_utc`, `status`, `details`
- On tick: evaluates cron expression → enqueues execution request → tracks run state

### 6.3 Integration

- Scheduler enqueues `"Scheduler"` request type into the execution queue (same pipeline as manual runs)
- Execution queue deduplication prevents overlapping runs
- Run state provides audit trail: when each scheduled run triggered, whether it succeeded/failed

## 7. Transform — NL to Logic/Conditional Graphs

### 7.1 NL → Transform Pipeline

**Files**: `transform.clj` (planner + validator), `transform/routes.clj` (HTTP), `transformChatComponent.js` (chat UI)

```
User NL -> parse-text (LLM or deterministic fallback) -> semantic plan
    -> semantic-plan->gil -> constrained GIL (linear chain)
    -> validate-transform-gil (topology + data flow)
    -> preview-data / preview-text -> UI preview
    -> apply-gil -> Bitool graph on canvas
```

- **`parse-text` / `resolve-plan`**: Tries Claude tool_use (`create_transform_graph`) first, validates generated GIL, attempts one structured repair pass on invalid/incomplete output, then falls back to `mock-spec` when LLM output is still unusable
- **`semantic-plan->gil`**: Generates linear node chain — source → Logic/Conditional stages → Response Builder → Output
- **`validate-transform-gil`**: Topology checks (1 source, 1 output, single linear chain), data flow validation (all `source_column` references must point to known upstream stage outputs)
- **Deterministic fallback library**: 5 domain templates — order/customer, invoice, lead, customer mastering, eligibility
- **Chat UI** (`<transform-chat>`): NL input → preview with validation status → "Apply Graph" button renders to canvas

### 7.2 GIL (Graph Intent Language)

GIL is the intermediate representation for LLM-proposed graph mutations.

**Pipeline**: NL → GIL document → normalize → validate → compile → apply to graph

**Implementation**: `bitool.gil.*` — `schema.clj` (type system), `normalize.clj`, `validator.clj`, `compiler.clj`, `api.clj` (HTTP handlers)

**Node type mapping**: `canonical-type` maps semantic names to btypes. `valid-edge?` enforces connectivity rules. Layout computed automatically (source-y=100, cascading right/down).

### 7.3 NL → Pipeline (Intent-to-Pipeline)

**Files**: `pipeline/intent.clj`, `pipeline/planner.clj`, `pipeline/compiler.clj`, `pipeline/discovery.clj`, `pipeline/reconcile.clj`, `pipeline/registry.clj`

```
User NL -> parse-intent (LLM + tool_use) -> PipelineIntent
    -> plan-pipeline (deterministic) -> PipelineSpec
    -> apply-pipeline! -> Bronze GIL + Silver proposals + Gold proposals
```

- **PipelineIntent schema**: `pipeline_name`, `source` (system, objects, auth, load_type, cadence), `target` (platform, catalog, schema), `bronze`, `silver`, `gold`, `ops`
- **Connector knowledge**: Pre-built registry of API specs, auth methods, pagination strategies, rate limits
- **Metric packages**: Predefined analytics patterns (conversion_funnel, cohort_analysis, etc.)

## 8. Model — Silver/Gold Semantic Layer

### 8.1 Silver Layer — Cleaned Business Entities

**Purpose**: Deduplicated, typed, business-logic-applied fact/dimension tables.

**Key files**: `modeling/automation.clj`, `compiler/core.clj`, `compiler/ir.clj`, `compiler/dialect/*.clj`

#### Proposal Lifecycle

```
propose-silver-schema!
    -> update-silver-proposal! (human edits)
    -> synthesize-silver-graph! (build graph)
    -> compile-silver-proposal! (graph -> SQL IR -> dialect SQL)
    -> validate-silver-proposal! (schema checks)
    -> validate-silver-proposal-warehouse! (sample execution)
    -> review-silver-proposal! (human approval)
    -> execute-silver-release! (run in warehouse)
```

- **Deterministic rules first**: Entity inference, column design rules, type casting
- **LLM assist**: Semantic ranking, explanation generation (partial)

### 8.2 Gold Layer — Analytics-Ready Marts

**Purpose**: Pre-aggregated, grain-optimized metric tables ready for reporting.

**Key files**: Same `modeling/automation.clj` with `propose-gold-schema!`, `compile-gold-proposal!`, `execute-gold-release!`

#### Gold Patterns
- Metric-catalog-driven mart generation
- Business intent → Gold model proposals
- Silver as source (Gold never reads Bronze directly)

### 8.3 Compiler (SQL Generation)

**Files**: `compiler/core.clj`, `compiler/ir.clj`, `compiler/dialect/*.clj`

```
Model Proposal -> compile-model -> SQL IR -> dialect-specific SQL
```

#### Supported Dialects

| Dialect | File | DDL | SELECT | Materialization |
|---------|------|-----|--------|-----------------|
| PostgreSQL | `dialect/postgresql.clj` | CREATE TABLE | SELECT | INSERT INTO ... SELECT |
| Databricks | `dialect/databricks.clj` | CREATE TABLE (Delta) | SELECT | MERGE INTO / INSERT |
| Snowflake | `dialect/snowflake.clj` | CREATE TABLE | SELECT | MERGE INTO / INSERT |
| BigQuery | `dialect/bigquery.clj` | CREATE TABLE | SELECT | MERGE / INSERT |

**IR** (`compiler/ir.clj`): Transforms proposal JSON into a dialect-independent SQL IR, then each dialect module renders it to native SQL.

## 9. Train — Schema Context & Auto-Training

**DB table**: `schema_context` — stores per-table and per-column business descriptions

```sql

  context_id BIGSERIAL PK,
  conn_id, schema_name, table_name, column_name (nullable = table-level),
```

## 10. Report — NL to Query to Chart

### 10.1 AI Reporting (ISL Pipeline)

**File**: `routes/home.clj` (reporting-* functions)

```
User question -> build-isl-prompt (registry + joins + schema context)
    -> nl->isl (LLM with tool_use) -> ISL document
    -> validate-isl (enum-locked, zero hallucination)
    -> compile ISL -> SQL
    -> execute against warehouse
    -> return results
```

- **ISL tool schema**: Table names are enum-locked (LLM must pick from available tables). Columns, aggregate references, and join keys are free-form strings validated server-side after generation (`valid-column-ref?`, `validate-join-spec`) — LLM can propose any column name but invalid references are rejected before SQL compilation
- **Schema context injection**: Business descriptions from training enrich the prompt
- **Conversational memory**: Connection-scoped, follow-up detection via `follow-up?` + `append-history-context`
- **Saved reports**: CRUD via `reporting-save`, `reporting-saved`, `reporting-run-saved`, `reporting-delete-saved`

### 10.2 Charting (Chart.js)

**File**: `reportingDashboard.js`

- Chart.js v4.4.4 via CDN, canvas-rendered
- 3 chart types: `lineChart()` (time-series), `barChart()` (categorical), `doughnutChart()` (distribution)
- `aiAutoChart()` — auto-detects chartable AI results (2-100 rows, date/string label column + numeric columns), offers "Visualize as chart" button
- `renderAIChart()` — smart type selection (date→line, string→bar), 8-color palette rotation, multi-dataset support
- `renderAIReport()` — main dashboard view with auto-chart (2-200 rows, 280px canvas)
- `GET /api/reporting/data` — pre-aggregated sample datasets (fuel, safety, utilization, compliance, etc.)

## 14. Testing — ELT/ETL Testing & Automated AI Testing

### 14.1 Philosophy

Every BI cycle stage has corresponding tests. Bitool tests the full ELT pipeline deterministically — from API extraction through schema inference, checkpoint management, batch manifests, and transformation proposals — without requiring a live warehouse. All AI-assisted features include deterministic fallback paths that are independently tested.

### 14.2 ELT/ETL Pipeline Testing

Tests validate the complete data movement pipeline:

| Test File | What it covers |
|-----------|---------------|
| `ingest_runtime_test.clj` (3800+ lines) | End-to-end runtime: endpoint config, bad record handling, checkpoint columns, batch manifests, adaptive backpressure, circuit breaker state |
| `ingest_execution_test.clj` (860+ lines) | Request queuing, execution orchestration, orphan recovery, lease heartbeats, concurrent request handling, status tracking |
| `ingest_checkpoint_test.clj` | Watermark query resolution, timestamp parsing, incremental state |
| `ingest_scheduler_test.clj` | Cron expression evaluation, scheduled run tracking |
| `execute_graph_test.clj` | Graph execution end-to-end — source → transform → target |
| `schema_infer_test.clj` | Field inference from JSON, OpenAPI→field descriptors, type detection, nullability |
| `schema_drift_test.clj` | Drift severity classification, DDL generation, type compatibility (widening/narrowing) |

**Warehouse mocking**: Tests mock JDBC interactions via `with-redefs`, enabling full pipeline validation without live warehouse dependencies.

### 14.3 Automated AI Testing

AI features are tested with dual-path verification — LLM path and deterministic fallback:

| Test File | What it covers |
|-----------|---------------|
| `ai_assistant_test.clj` (600+ lines) | 10+ AI task types: schema explanation, key suggestion, proposal explanation, validation failure analysis, Silver transform generation, BRD-to-proposal, drift explanation, remediation, metric glossary, KPI anomaly explanation |
| `ai_llm_test.clj` | Provider switching (Anthropic→OpenAI), API key validation, parameter passing |
| `routes/home_test.clj` | ISL pipeline: NL→ISL→SQL→results, schema context, fingerprint detection, auto-train with curated vs LLM paths |
| `pipeline_compiler_test.clj` | NL→pipeline compilation, Silver/Gold proposal generation |
| `transform_routes_test.clj` | NL→GIL validation errors, end-to-end NL→transform, apply + session handling |
| `gil_test.clj` | GIL compile, validate, normalize |

**Deterministic fallback testing**: Every AI assistant function has a fallback mode (LLM unavailable) that returns heuristic responses with warnings. Tests verify both paths independently — ensuring the system never hard-fails on LLM outage.

**Confidence calibration**: AI responses include normalized confidence scores (0-1 scale). Tests verify that deterministic fallbacks return appropriate low-confidence markers.

## Appendix A: Graph Data Model

All data pipelines are represented as directed acyclic graphs of typed nodes.

### A.1 Graph Structure

```clojure
{:a {:name "pipeline-name" :v 3 :id 42}   ;; attributes
 :n {node-id {:na {attrs} :e {target-id {}}}}}  ;; nodes + edges
```

- **`:a`** — graph attributes (name, version, id)
- **`:n`** — nodes keyed by integer ID. Each node has `:na` (attributes including `:b` btype) and `:e` (outgoing edges to child node IDs)
- **Output node** — always btype `"O"`, found via `find-node-id-by-btype g "O"`
- **`connected-graph`** — filters to nodes reachable from Output via `predecessors`. Orphans go to `sp`.

### A.2 Node Types (btype codes)

| Code | Type | Purpose | Files |
|------|------|---------|-------|
| T | Table | Database table source | tableComponent.js |
| Ep | Endpoint | REST API source | apiComponent.js, endpoint.clj |
| Wh | Webhook | Inbound webhook source | webhookComponent.js |
| J | Join | SQL JOIN (inner, left, right, full, cross) | joinEditorComponent.js |
| P | Projection | Column selection/rename | projectionComponent.js |
| Fi | Filter | WHERE clause | filterComponent.js |
| A | Aggregation | GROUP BY + aggregate functions | aggregationEditorComponent.js |
| S | Sorter | ORDER BY | sorterEditorComponent.js |
| U | Union | UNION / UNION ALL | unionEditorComponent.js |
| Fu | Function | Column-level transform expressions | functionComponent.js |
| Tg | Target | Destination warehouse table | targetComponent.js |
| Mp | Mapping | Column mapping (Bronze→Silver) | mappingEditorComponent.js |
| O | Output | Terminal output node | (implicit) |
| Au | Auth | Authentication config | authComponent.js |
| Vd | Validator | Input validation rules | validatorComponent.js |
| Rl | RateLimiter | Request throttling | rateLimiterComponent.js |
| Cr | CORS | Cross-origin config | corsComponent.js |
| Lg | Logger | Request/response logging | loggerComponent.js |
| Cq | Cache | Response caching | cacheComponent.js |
| Ev | EventEmitter | Event dispatch | eventEmitterComponent.js |
| Ci | CircuitBreaker | Fault tolerance | circuitBreakerComponent.js |
| Sc | Scheduler | Cron-based scheduling | schedulerComponent.js |
| Dx | DbExecute | Raw SQL execution | dbExecuteComponent.js |
| Rb | ResponseBuilder | Custom response formatting | responseBuilderComponent.js |

### A.3 Edge Rules

`rectangles` map in `graph2.clj` defines allowed parent→child connectivity per btype. Example: `"T" ["J" "U" "P" "A" "S" "Fi" "Fu" "Tg" "O"]`.

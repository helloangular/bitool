JSON Explode Patterns

How API responses become Bronze rows and Silver tables in BiTool
User Guide v1.0 · BiTool Medallion Architecture

The Big Picture

REST APIs return JSON. BiTool's JSON Explode engine extracts records from that JSON, writes each as a row into Bronze (raw storage), and then flattens/splits them into Silver tables (clean, queryable).

🔎

Explode Rule

A JSON path (e.g. data) telling BiTool where the array of records lives inside the API response.

🧩

Bronze = Raw Rows

One table per endpoint. Each array element becomes a row. The full JSON record is stored in payload_json. Promoted columns are extracted alongside.

🔬

Silver = Clean + Split

Scalar fields are flattened into columns. Nested child arrays (e.g. line_items[]) become separate Silver tables, joined by business key.

🏆

Gold = Aggregated

Fact tables get time-grain aggregation (daily, weekly). Dimension tables get summary views. Built from Silver.

💡
Key Rule: The explode rule only controls which array to "unpack" at ingestion time. Everything below that array stays nested in Bronze. The splitting into multiple tables happens at the Silver layer, not Bronze.

Most Common

1Simple Array

The API wraps records inside a named array. This is the most common REST API pattern — used by Samsara, Stripe, Salesforce, and most modern APIs.

API Response
GET /api/vehicles
Returns { "data": [...] }
Explode Rule
path = "data"
Normalized to $.data[]
Bronze
1 row per vehicle
3 vehicles = 3 rows
Silver
silver_vehicles
Flat columns, clean types

API Response JSON

Response { "status": "ok", "total": 3, "data": [ ← explode here { "id": "V001", "name": "Truck Alpha", "miles": 84200, "status": "active" }, { "id": "V002", "name": "Van Beta", "miles": 31500, "status": "active" }, { "id": "V003", "name": "Sedan Gamma", "miles": 12750, "status": "maintenance" } ] }
ⓘ Config: Set json_explode_rules path to data. BiTool normalizes it to $.data[] and iterates over each item.

Bronze Table: bronze_vehicles

ingestion_id run_id data_items_id data_items_name payload_json
ing_001 run_17 V001 Truck Alpha {"id":"V001","name":"Truck Alpha", ...}
ing_001 run_17 V002 Van Beta {"id":"V002","name":"Van Beta", ...}
ing_001 run_17 V003 Sedan Gamma {"id":"V003","name":"Sedan Gamma", ...}

Silver Table: silver_vehicles

id name miles status ingested_at_utc
V001 Truck Alpha 84200 active 2026-03-27 10:00
V002 Van Beta 31500 active 2026-03-27 10:00
V003 Sedan Gamma 12750 maintenance 2026-03-27 10:00
📌
What happened: The "data" array had 3 objects. BiTool created 3 Bronze rows (one per object) and 1 Silver table with flattened columns. Metadata fields (status, total) outside the array are ignored — only the records inside the explode path are ingested.

4Parent-Child (Nested Arrays)

Each record contains a nested array of child objects — e.g., an invoice with line items, an order with products, a trip with stops. This is where Bronze and Silver diverge significantly.

API Response
GET /api/invoices
Each invoice has line_items[]
Explode Rule
path = "data"
Explode at invoice level
Bronze
1 row per invoice
line_items nested inside
Silver
2 Silver tables
Invoices + Line Items

API Response JSON

Response { "data": [ { "invoice_id": "INV-001", "vendor": "Acme Corp", "total": 1500.00, "date": "2026-03-15", "line_items": [ ← nested child array { "sku": "WDG-100", "desc": "Widget", "qty": 10, "price": 100.00 }, { "sku": "GDG-200", "desc": "Gadget", "qty": 5, "price": 100.00 } ] }, { "invoice_id": "INV-002", "vendor": "Beta LLC", "total": 800.00, "date": "2026-03-20", "line_items": [ { "sku": "SRV-300", "desc": "Service", "qty": 1, "price": 800.00 } ] } ] }
🔄
The Split: Bronze stores the entire invoice (including nested line_items) in one row. Silver creates two tables — one for invoices (parent) and one for line items (child), linked by invoice_id as the foreign key.

Complex

5Multi-Level Nesting

Three or more levels of nested arrays: warehouses contain zones, zones contain racks, racks contain items. BiTool supports up to 8 levels of depth.

API Response JSON

Response { "data": [ { "warehouse_id": "WH-1", "city": "Chicago", "zones": [ ← level 1 child { "zone_id": "Z-A", "temp": "cold", "racks": [ ← level 2 child { "rack_id": "R-01", "capacity": 500, "items": [ { "sku": "ICE-1", "qty": 120 } ] } ] } ] } ] }
Bronze
1 table
1 row per warehouse
zones, racks, items nested in payload_json
➔
Silver
silver_warehouses
warehouse_id, city
silver_warehouse_zones
warehouse_id, zone_id, temp
silver_zone_racks
zone_id, rack_id, capacity
silver_rack_items
rack_id, sku, qty
⚠ Important: The explode rule is still data (the top-level array). BiTool does NOT explode nested arrays at Bronze. All nesting is preserved. The grain planner detects the multi-level structure and recommends separate Silver tables for each level.
Advanced

6Peer Arrays (Sibling Arrays)

A single record contains multiple arrays at the same level — e.g., a product with both tags[] and reviews[]. Each peer array becomes its own Silver table.

Common

7Paginated / Cursor Responses

APIs return data in pages with a cursor or next-page token. BiTool fetches all pages, applies the same explode rule to each page, and merges all records into one Bronze table.

Page 1

{ "data": [ { "id": 1, "name": "Alpha" }, { "id": 2, "name": "Beta" } ], "pagination": { "next_cursor": "abc123", "has_more": true } }

Page 2

{ "data": [ { "id": 3, "name": "Gamma" } ], "pagination": { "next_cursor": null, "has_more": false } }
ⓘ How it works: BiTool fetches pages until has_more is false (or cursor is null). The explode rule data is applied to each page independently. All extracted records go into the same Bronze table.

Bronze Table: bronze_items

ingestion_iddata_items_iddata_items_namepayload_json
ing_0071Alpha{"id":1,"name":"Alpha"}
ing_0072Beta{"id":2,"name":"Beta"}
ing_0073Gamma{"id":3,"name":"Gamma"}
All 3 records from 2 pages land in the same table. The pagination metadata is not stored — only the records.

Silver Table: silver_items

idnameingested_at_utc
1Alpha2026-03-27 10:00
2Beta2026-03-27 10:00
3Gamma2026-03-27 10:00
Edge Case

8Single Object (No Array)

The API returns a single object — not wrapped in an array. This is typical for "get by ID" endpoints or config/settings endpoints. BiTool treats the entire response as one record.

API Response JSON

Response { "org_id": "ORG-999", "name": "Acme Industries", "plan": "enterprise", "seats": 250, "created_at": "2024-01-15T00:00:00Z" }
ⓘ Config: Leave explode path blank. BiTool detects this is a single object (not an array) and wraps it as a single-record array.

Bronze Table: bronze_org

ingestion_iddata_items_org_idpayload_json
ing_008ORG-999{"org_id":"ORG-999","name":"Acme Industries", ...}
Always exactly 1 row per API call.

Silver Table: silver_org

org_idnameplanseatscreated_at
ORG-999Acme Industriesenterprise2502024-01-15

Quick Reference

All 8 patterns at a glance — when to use each configuration and what to expect.

# Pattern API Shape Explode Path Bronze Rows Silver Tables
1 Simple Array {"data": [...]} data 1 per array element 1 table
2 Root Array [{...}, {...}] (empty) 1 per array element 1 table
3 Deep Nested {resp: {results: {items: [...]}}} response.results.items 1 per item 1 table
4 Parent-Child {data: [{..., items: [...]}]} data 1 per parent 2 tables (parent + child)
5 Multi-Level zones[].racks[].items[] data 1 per top entity N tables (one per level)
6 Peer Arrays {..., tags:[], reviews:[]} data 1 per parent 1 parent + 1 per array
7 Paginated {data:[], next_cursor:""} data All pages combined 1 table
8 Single Object {...} (empty) 1 row 1 table

Bronze Table Anatomy

Every Bronze table has the same envelope structure regardless of pattern. The explode rule only determines how many rows are created and what goes into each row.

Column Type Description
ingestion_id STRING Unique ID for this ingestion run
run_id STRING Execution run identifier
ingested_at_utc TIMESTAMP When this row was ingested
payload_json STRING Full JSON record (one element from the exploded array)
data_items_* varies Promoted columns — scalar fields extracted from the record for fast querying without parsing JSON
📚
Promoted columns are a subset of the record's scalar fields that are extracted into real table columns alongside payload_json. They enable efficient SQL queries on Bronze without JSON parsing. The schema inferrer selects the best candidates (IDs, timestamps, business keys).

payload_json always contains the complete record — including nested arrays and objects. Nothing is lost.

Tips & Best Practices

Test with Preview First

Use the Preview Schema button before running a full ingestion. It fetches one page, applies the explode rule, and shows you exactly what the Bronze columns will look like.

Let the Grain Planner Help

After schema inference, BiTool's grain planner analyzes nested arrays and recommends Silver table splits, primary keys, and watermark columns automatically.

Promote Only Key Columns

Don't promote every field to Bronze columns. Focus on IDs, timestamps, and business keys. Everything else is in payload_json and gets flattened at Silver.

Path Format is Forgiving

You can enter data, $.data, or $.data[] — BiTool normalizes all three to the same canonical form. Use whatever feels natural.

Don't Explode Too Deep

Set the explode path to the main records array, not a deeply nested child. Let Silver handle the sub-arrays. Exploding at the wrong level creates too many Bronze rows with sparse data.

Pagination is Transparent

The explode rule applies per-page. Whether the API returns 1 page of 1000 records or 100 pages of 10, the Bronze table looks the same.

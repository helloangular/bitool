BiTool Input → Bronze → Silver → Gold Architecture

This document maps the current codebase and design into one end-to-end view: API, Kafka stream, Kafka consumer, file, and mainframe file ingestion into Bronze, then Bronze to Silver and Gold modeling, review, compilation, and execution. It also calls out where resilience, idempotency, concurrency control, and vendor-specific logic live, and whether the architecture is generic enough for BigQuery.

  • Snowflake load history
  • Databricks jobs / SQL

Airflow orchestrates around Bitool; it does not replace Bitool’s queue, checkpoint, manifest, or replay engine.
Concern Best Owner Reason
Scheduled batch kickoff Airflow or enterprise scheduler Good fit for time-based orchestration and cross-team dependency chains.
Kafka stream lifecycle and offset safety Bitool Requires source-aware runtime semantics, checkpoints, and post-commit offset handling.
Bronze queue / worker leases / orphan recovery Bitool These are internal execution semantics tied to manifests, retries, and replayability.
Replay / rollback / bad-record reprocessing Bitool These actions depend on domain-specific batch state, not generic DAG orchestration.
Cross-product workflows Airflow Useful when Bitool is one step among many systems.
Schema review / Silver / Gold publish lifecycle Bitool These are part of the medallion control plane and compiler workflow.

Recommended split: Bitool is the ingestion and medallion control plane; Airflow is an optional outer orchestrator. Airflow should call Bitool, not reimplement its queue, stream runtime, checkpointing, or replay semantics.

8. Is the Architecture Generic Enough for BigQuery?

Short answer: mostly yes at the control-plane and compiler-IR layers, but not yet plug-and-play. BigQuery is feasible without redesigning the whole product, but it still needs a new warehouse adapter and some refactoring.

Already Generic Enough

  • Graph/source node model
  • Bronze manifest / checkpoint / replay concepts
  • Silver / Gold proposal lifecycle
  • Compiler IR
  • Modeling console and review workflow

Needs Refactor or New Adapter Work

  • New compiler/dialect/bigquery.clj
  • BigQuery execution / validation packaging
  • BigQuery-specific DDL and load path in db.clj
  • Warehouse execution routing in modeling automation
  • Physical load semantics for Bronze bulk ingest
Area BigQuery Fit Refactor Needed?
Bronze control-plane semantics Good No redesign
Silver / Gold proposal model Good No redesign
Compiler dispatch Good Add dialect backend
Warehouse execution routing Partial Yes; more adapter isolation needed
Vendor-specific physical load path Not present Yes; BigQuery load jobs would need a new block
Databricks/Snowflake assumptions in some runtime code Manageable Yes; continue adapter extraction

Conclusion: the product is architecturally extensible to BigQuery, but not “add one connection string and it works”. It needs a BigQuery vendor adapter at the dialect, load, validation, and execution-packaging layers.

9. Practical Takeaway

The architecture is already split the right way:

  • The control plane is mostly generic: graph config, Bronze runtime model, modeling workflow, IR, review, releases, observability, and governance.
  • The data plane / warehouse adapter layer is where vendor-specific code belongs: SQL dialect, DDL/load path, warehouse validation, and execution packaging.
  • The system is strong enough to support Databricks and Snowflake today, and can support BigQuery next if the current warehouse adapters continue to be extracted cleanly.
